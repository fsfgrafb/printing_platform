# 部署说明

## 目标环境

- Windows 10 IoT 企业版 LTSC 21H2
- 固定 IP：`10.18.47.101`
- 打印机队列：`HP LaserJet Professional P1106`，USB 连接
- 前端监听：`0.0.0.0:80`
- 后端监听：`127.0.0.1:8080`

## 运行流程

1. 构建后端：

   ```powershell
   cargo build --release
   ```

2. 构建前端：

   ```powershell
   cd frontend
   npm install
   npm run build
   cd ..
   ```

3. 保持 `frontend/dist` 在项目目录内。项目默认由前端服务占用 `80` 端口，并把 `/api` 请求代理到后端 `127.0.0.1:8080`。

4. 修改 `config.toml`：

   ```toml
   [server]
   bind = "127.0.0.1:8080"

   [printer]
   name = "HP LaserJet Professional P1106"
   simulate = false
   command_timeout_seconds = 60
   backend_script = "scripts/printer-backend.ps1"
   job_discovery_seconds = 20
   pdf_printer_path = ""

   [converter]
   office_program = "powershell"
   office_args = ["-NoProfile", "-ExecutionPolicy", "Bypass", "-File", "scripts/convert-office.ps1", "-InputPath", "{input}", "-OutputPath", "{output}"]
   command_timeout_seconds = 180
   ```

5. 首次启动后，用 `initial_admin_student_id` 登录，默认密码同学号。

6. 启动后端：

   ```powershell
   .\target\release\print-server.exe
   ```

7. 另开终端启动前端：

   ```powershell
   cd frontend
   npm run preview
   ```

8. 浏览器访问：

   ```text
   http://10.18.47.101
   ```

如果 PowerShell 中提示找不到 `node` 或 `npm`，但 Node.js 安装在默认位置，可先执行：

```powershell
$env:PATH='C:\Program Files\nodejs;' + $env:PATH
```

## Windows 服务

建议使用 NSSM 或同类服务包装器运行 `target/release/print-server.exe`，并指定工作目录为项目根目录。Office COM 转换通常需要服务运行在一个已安装并激活 Microsoft 365 的普通用户账号下。

前端可使用 `npm run preview` 临时运行在 `80` 端口。长期运行建议使用 IIS、Nginx 或其他静态文件服务器托管 `frontend/dist`，并配置反向代理：

```text
/api -> http://127.0.0.1:8080/api
```

## 打印策略

真实打印由 `scripts/printer-backend.ps1` 提交，并在提交后捕获 Windows 作业 ID。推荐在目标机为所有用户安装 SumatraPDF；也可以把便携版放在项目的 `tools/SumatraPDF.exe`，或通过 `printer.pdf_printer_path` 指定路径。后端也会探测 Adobe Reader，最后才尝试系统 PDF `PrintTo` 关联。部署账户必须能够查询目标队列的作业。

调度器每 5 秒读取一次 `.NET System.Printing` 的具体状态和 PrintManagement 作业列表：

- 缺纸、卡纸、脱机、队列暂停、舱门打开等状态会临时阻止提交后续任务，故障恢复后自动继续。
- 墨粉不足只产生可确认的管理员告警，不阻塞任务。
- 已观察到的 Windows 作业从队列消失后，平台记录“Windows 打印作业已结束”并结算页数；这不等同于硬件提供了物理出纸确认。
- 提交失败或无法捕获作业 ID 时，平台会设置管理员暂停，防止重复提交。

正式启用前必须用一个普通 PDF 完成端到端测试。推荐的 SumatraPDF 调用使用 `-print-to` 和 `simplex,monochrome,fit`，不依赖当前账户的 PDF 文件关联。

## 数据保留

SQLite 数据库和上传文件默认位于 `data/`。后台清理任务默认每 6 小时执行一次：

- 24 小时前仍未提交的临时上传会被删除。
- 365 天前已完成或已取消的任务记录及对应文件会被删除。

打印队列页面同时承担历史功能。最终筛页、渲染后的 PDF 在任务结束后继续保留，直到上述 365 天保留期到期时与记录一起删除。所有登录用户都可在打印队列中查看记录并预览最终 PDF。

这些时间可通过 `config.toml` 中的 `cleanup_interval_hours`、`file_retention_days`、`temp_upload_retention_hours` 调整。

## 运维检查

- `GET http://127.0.0.1:8080/api/health`：在打印主机本机检查数据库、队列暂停状态和打印机模式。
- `GET http://10.18.47.101/api/health`：通过前端 80 端口代理检查后端状态。
- `GET http://10.18.47.101/api/admin/stats.csv`：管理员登录后可导出统计报表。

## 页面状态说明

打印队列向所有登录用户展示打印机名称和状态：

- `Ready`：可用、空闲（通常对应驱动的 `Normal`）。
- `Running`：初始化、处理中或忙碌（如 `Active`、`Processing`、`Busy`、`Initializing`、`Waiting`）。
- `Printing`：驱动报告正在打印。
- `Paused`：管理员暂停平台队列。
- `Offline`：打印机不可用或脱机。
- `Error`：缺纸、卡纸、舱门打开或其他阻塞错误。

页面同时显示驱动原始状态便于排障。`TonerLow`/`NoToner` 属于管理员告警，不会自动阻塞队列。
