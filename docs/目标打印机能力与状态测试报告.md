# 目标打印机能力与状态测试报告

## 1. 测试对象

- 目标系统：Windows 10（内部版本 19044），64 位
- PowerShell：5.1
- 运行账户：`ACM`
- 账户权限：Windows 管理员账户
- 打印机队列名称：`HP LaserJet Professional P1106`
- 驱动：`HP LaserJet Professional P1106`
- 连接端口：`USB001`
- 连接方式：本地 USB 打印机
- Print Spooler：正在运行，启动类型为自动

## 2. 已确认的物理能力

- 打印机只支持黑白打印，不支持彩色打印。
- 打印机完全不具备自动双面打印能力。
- 如需双面打印，只能由人工翻面后再次打印；无人值守服务不得把驱动列出的双面选项当作硬件自动双面能力。

## 3. 静态探测结果

目标系统可以通过以下三条通道访问打印机：

1. Windows PrintManagement：可以查询打印机、默认配置和打印作业。
2. CIM/WMI `Win32_Printer`：可以发现队列并读取部分粗粒度状态。
3. .NET `System.Printing`：可以读取队列状态、作业和打印能力。

当前能够调用或查询的主要内容包括：

- 队列是否正常、离线、暂停、忙碌、正在打印或发生错误。
- 是否需要用户干预、缺纸、卡纸、低墨粉、门打开或要求手动进纸。
- 当前队列作业数量，以及作业编号、名称、状态、总页数和已打印页数。
- 当前打印配置，包括黑白、单面、纸张和方向。
- 驱动声明的纸张尺寸、方向、份数和其他打印票据能力。

静态探测时队列状态为正常/空闲，队列中没有打印作业。

## 4. 数据可信度判断

### 可直接采用

- 队列名称、驱动、端口和 Spooler 状态。
- PrintManagement 返回的队列状态和作业列表。
- .NET `System.Printing` 返回的队列状态标志和作业列表。
- 黑白、单面等当前打印配置。

### 只能作为参考

- CIM/WMI 的 `Capabilities` 同时声明了彩色和双面，与打印机实际硬件不符，因此不能用它决定业务打印选项。
- 驱动打印能力中出现长边和短边双面选项，但打印机没有自动双面硬件。这些选项可能对应驱动的人工双面流程，服务端必须忽略。
- 缺纸、卡纸、低墨粉等字段虽然可以调用，但只有在制造对应故障并提交真实作业后，才能确认驱动是否会实时上报。

## 5. 项目接入原则

- 生产配置必须使用完整队列名称 `HP LaserJet Professional P1106`。
- 平台只提供黑白、单面打印，不向用户展示彩色或双面选项。
- 主要状态源采用 .NET `System.Printing`，PrintManagement 作为作业查询和状态备用来源。
- CIM/WMI 只用于打印机发现和粗粒度兜底，不使用其能力列表决定打印选项。
- 缺纸、卡纸、离线、暂停、严重错误或需要人工干预时，平台应停止向 Windows 队列继续提交新任务。
- 低墨粉可作为告警；是否阻止新任务应根据真实故障测试结果决定。
- 打印命令返回成功只代表作业已进入 Windows 队列，不能代表纸张已经打印完成。平台应继续跟踪 Windows 作业状态。
- 测试作业必须在采样完成后从队列删除，避免重新装纸或墨粉后意外输出。

## 6. 动态故障测试

### 6.1 测试条件

- 打印机已移除全部纸张和墨粉。
- 通过 Windows GDI 向真实队列提交一页测试作业。
- 采样时间为 30 秒，每两秒采样一次，共 15 次。
- 测试作业名称：`Printing Platform Probe 231b7ccff97242e08fd226e34a7b4f30`
- Windows 作业 ID：`3`

### 6.2 状态变化

- 作业于 2026-07-10 13:10:59 成功提交。
- 前 4 次采样（约 8 秒）打印机仍显示 `Normal`，说明硬件错误上报存在延迟。
- 从约第 10 秒开始，后续 11 次采样均稳定显示 `PaperOut`。
- PrintManagement 的打印机状态变为 `PaperOut`。
- .NET `System.Printing.QueueStatus` 变为 `PaperOut`，`IsOutOfPaper` 变为 `true`。
- 作业在全部 15 次采样中均为 `Printing, Retained`，页数始终为 `0/1`，没有被误判为完成。
- 队列中始终存在 1 个作业。

### 6.3 未上报的状态

当时认为已经移除墨粉，但后来确认硒鼓内仍有残余墨粉；同时这两轮还存在缺纸这一优先阻塞条件。因此只能确认“缺纸状态下未观察到缺墨提示”，不能据此判断驱动是否支持墨粉告警。

缺纸期间以下通用状态仍为 `false`：

- `IsInError`
- `NeedUserIntervention`
- `IsBusy`
- `IsPrinting`
- `IsProcessing`

因此不能只检查通用错误或人工干预标志。必须直接检查 `QueueStatus` 和 `IsOutOfPaper` 等具体状态。作业自身显示 `Printing` 时，队列的 `IsPrinting` 仍可能为 `false`，两者含义不同。

### 6.4 清理结果与安全事项

旧版脚本调用 `Remove-PrintJob` 后，作业 ID 3 在一次短暂复查中仍然存在，报告中的 `cleanup_succeeded` 为 `false`。在重新装入纸张前，必须在 Windows 打印队列窗口确认并删除该测试作业。

探测脚本现采用四级清理和最长 10 秒复查：

1. `Remove-PrintJob`
2. `.NET System.Printing` 的作业 `Cancel()`
3. CIM `Remove-CimInstance`
4. Win32 Spooler `SetJob(JOB_CONTROL_DELETE)`

第二轮测试进一步证明 `Remove-PrintJob` 和 `.NET Cancel()` 都未能立即移除处于 `Printing, Retained` 状态的作业，CIM 类也没有暴露可调用的 `Delete` 方法。脚本现已改为使用 `Remove-CimInstance`，并增加 Win32 Spooler `SetJob(JOB_CONTROL_DELETE)` 作为最终兜底。

第二轮测试作业 ID 4 也仍可能留在目标机队列。在重新装入纸张前，必须确认作业 ID 3 和 4 均已从 Windows 打印队列消失。

### 6.5 第二轮动态测试

第二轮作业于 2026-07-10 13:14:57 提交，作业 ID 为 `4`，共采样 15 次：

- 第 1 次采样为 `PaperOut`，`IsOutOfPaper=true`。
- 此时计算机弹出缺纸提示，操作人员点击了“继续”。
- 点击“继续”后，随后 5 次采样为 `Initializing`，`IsInitializing=true`。
- 再之后 9 次采样为 `Printing`，`IsPrinting=true`，打印机发生了一次空转。
- 作业在全部采样中仍是 `Printing, Retained`，已打印页数始终为 `0/1`。
- `IsTonerLow` 在全部采样中仍为 `false`。

因此，`PaperOut → Initializing → Printing` 是人工点击“继续”所触发的驱动恢复/重试流程，不能解释为驱动在无人操作时自行清除了缺纸状态。该流程同时证明：点击“继续”后出现 `Printing` 和打印机空转，也不代表纸张已经输出或作业已经完成。

### 6.6 动态测试结论

- 缺纸状态可以通过 PrintManagement 和 `.NET System.Printing` 可靠查询。
- 缺纸上报约有 10 秒延迟，调度器应持续轮询，不能在提交命令成功后立即判定打印完成。
- 不同测试中缺纸可能立即上报，也可能约 10 秒后上报。
- 操作人员点击系统缺纸提示中的“继续”会让状态进入 `Initializing` 和 `Printing`；平台应记录这种人工操作造成的状态转换。
- 缺墨粉能力尚未得到独立验证：缺纸可能阻止打印机进入检测或上报墨粉故障的阶段。
- 应以 Windows 作业是否仍存在、已打印页数及最终作业结果判断任务进度。
- `Printing` 只表示 Spooler/驱动正在处理或重试，不能作为实际打印进度或成功依据。
- `PaperOut` 必须阻止继续提交后续平台任务，并提示管理员补纸。

## 7. 第三轮：有纸、硒鼓有残余墨粉测试

### 7.1 测试条件

- 测试开始前 Windows 打印队列为空。
- 打印机已装入纸张；测试前认为没有墨粉，但测试后确认硒鼓内仍有残余墨粉。
- 采样期间未操作系统提示。
- 采样时间为 60 秒，每两秒一次，共 30 次。
- 测试作业 ID 为 `6`。

在本轮开始前曾观察到上一轮作业处于“正在删除”状态，手动取消也显示失败；重启打印机后队列恢复。这表明之前的作业清理请求可能已经被 Spooler 接受，但驱动或设备端未能完成删除，直到打印机重启才解除。不能把这种情况简单解释为当前账户权限不足。

### 7.2 状态时间线

- 作业于 2026-07-10 13:26:16 成功提交。
- 前 3 次采样中队列状态为 `Printing`，作业为 `Printing, Retained`，已打印页数为 `0/1`。
- 约 6 至 8 秒后，作业自动从 Windows 队列消失。
- 后续 27 次采样中，队列一直为 `Normal`，作业数为 0。
- 30 次采样中的 `IsTonerLow`、`IsInError` 和 `NeedUserIntervention` 全部为 `false`。
- 打印机成功输出了一张纸，纸上有探测程序绘制的几个单词。
- 探测页本来只包含一行短文本，因此“只有几个单词”是预期内容，不是打印残缺的证据。
- 没有出现低墨粉或无墨粉提示。

### 7.3 清理结果

报告中的 `cleanup_succeeded=true`，测试结束后队列正常为空。脚本执行清理时作业已经自动消失，因此 `Remove-PrintJob` 返回“指定的作业不存在”；这不是清理失败。脚本已调整为在删除前重新检查作业是否存在，并将这种情况记录为 `AlreadyAbsent`。

本轮没有实际触发后续的 `.NET Cancel()`、CIM 或 Win32 `JOB_CONTROL_DELETE`，所以强制删除兜底仍未经过目标机实测。

### 7.4 本轮测试结论

- 本轮是一次物理打印成功的正常路径测试，不是真正的无墨粉测试。
- 成功路径为：作业短暂显示 `Printing, Retained`，随后从队列消失，队列恢复 `Normal`，同时纸张实际输出。
- `IsTonerLow=false` 说明驱动没有把当前残余墨粉量识别为低墨，但不能证明驱动永远无法上报低墨或无墨。
- 在本次样本中，Windows 作业消失与物理打印成功一致；平台仍不能仅凭一次样本把“作业消失”等同于每次都物理成功。
- 真正的墨粉耗尽状态未完成实测，后续按既往弹窗记录作为管理员提示能力处理。

## 8. 最终可用状态边界

经过四轮测试，可以确定：

- 缺纸：能够查询，但可能有数秒延迟；`PaperOut`/`IsOutOfPaper` 可用于暂停平台队列。
- 低墨粉/无墨粉：本次探测未捕获状态，但目标机过去曾弹出一次估算型缺墨警告。产品按“支持提示”处理，但不将其视为可靠的硬件故障信号。
- Windows 作业状态：可以查询。本轮正常打印中，作业消失与实际输出一致，但仍不能单独作为绝对的物理成功证明。
- 作业清理：通常可以请求删除，但驱动/设备卡住时可能长期显示“正在删除”，必要时需要重启打印机或 Print Spooler。
- 平台完成状态应表述为“已提交至打印机/Windows 作业已结束”，不应表述为“已确认成功打印”。

## 9. 第四轮：5 页约 20% 覆盖率测试

### 9.1 测试条件

- 测试开始前队列为空。
- 硒鼓有少量残余墨粉。
- 单个作业包含 5 页，每页约 20% 黑色覆盖率。
- 采样 60 秒，每两秒一次，共 30 次。
- 测试作业 ID 为 `7`。

### 9.2 状态时间线

- 作业于 2026-07-10 13:36:29 成功提交。
- 前 10 次采样、约 20 秒内，打印机和队列均显示 `Printing`。
- 作业状态始终为 `Printing, Retained`，Windows 上报的页数一直是 `0/5`。
- 约第 22 秒开始作业从队列消失，后续 20 次采样均为 `Normal`，队列为空。
- `IsTonerLow`、`IsInError`、`NeedUserIntervention` 等所有告警标志均未触发。
- 清理结果为 `cleanup_succeeded=true`；清理阶段作业已经消失，记录为 `AlreadyAbsent:7`，没有警告或错误。

本轮再次证明驱动的 `PagesPrinted` 不可用于实际页数统计：即使作业包含 5 页并完成处理，该字段仍保持为 0。

## 10. 缺墨提示的产品约定

不再继续消耗墨粉进行测试。根据目标机过去曾弹出一次缺墨警告的人工观察，平台按以下约定实现：

- 将 `IsTonerLow=true`，或 `QueueStatus` 中出现 `TonerLow`、`NoToner` 等状态，视为缺墨提示。
- 缺墨只向管理员展示告警，不暂停平台队列，不取消当前作业，也不自动判定作业失败。
- 告警文案应说明这是驱动估算结果，例如“打印机可能墨粉不足，请管理员检查硒鼓”。
- 同一次告警应去重，避免轮询期间反复弹窗；管理员关闭或确认后不重复打扰，除非状态先恢复正常后再次出现。
- `IsTonerLow=false` 不代表墨粉一定充足。
- 这一能力属于产品假设和管理员辅助提示，不属于四轮探测中已稳定复现的硬件状态。
