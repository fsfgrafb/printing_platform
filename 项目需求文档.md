| 项目       | 内容                                                         |
| ---------- | ------------------------------------------------------------ |
| 项目名称   | ACM实验室自助打印平台                                        |

### 1. 打印功能

- **支持的文件格式**  
  优先级：PDF、Word (.docx) → 后期扩展 Excel、常见图片格式 (.jpg/.png/.bmp) → 小概率 txt/pptx。
- **文档转换流程**  
  所有上传文件在服务端统一转换为 PDF：
  - Word/Excel/PPT：通过 Microsoft 365 命令行或 COM 自动化导出为 PDF。
  - 图片：使用 Rust 库或命令行工具（如 ImageMagick）拼接为 PDF。
  - 文本文件：可选转为简单 PDF。
  - 转换后的 PDF 作为**预览文件**和**最终打印文件**，确保预览与打印完全一致。
- **打印调用方式**  
  通过 Windows Shell 命令或 Win32 API 将 PDF 直接发送到 HP LaserJet P1106 打印队列（如 `print /d:"HP LaserJet P1106" "file.pdf"`）。
- **单双面处理**  
  - 不支持自动双面打印。但提供**手动双面辅助**：用户上传文件后可选择“打印奇数页”或“打印偶数页”。选择后，系统重新生成仅含指定页面的 PDF，覆盖原预览文件并提交打印。双面打印时，一张物理纸的两面算**2张**限额，也就是说额度根据页数定而不是张数。
- **单日打印限额**  
  - 全局默认限额：**50页/人/天**。
  - 超出限额时：
    - 用户仍可提交，任务状态变为“待审核”。
    - 提交时二次确认，并显示管理员联系方式（QQ、学号，管理员需在设置中填写）。
    - 管理员可在管理面板同意或拒绝该任务，同意的任务不会占用对应用户当天的限额。
    - 次日零点**不自动刷新待审核任务**，任务保持待审核，除非管理员手动通过或限额恢复后重新提交。
  - 限额计数基于**当天所有已完成的打印任务页数之和**（含双面的两面各算一张）。
- **上传限制**  
  - 单文件大小、一次上传数量均无限制。
  - 建议前端做体积警告，但不断开。服务端需处理大文件，保存到临时目录。

### 2. 打印队列

- **队列模型**  
  - 每个上传的文件视为一个独立打印任务。
  - 任务状态：`queued`（队列中）、`printing`（打印中）、`done`（已打印）、`cancelled`（已取消）、`pending_review`（待审核）。
  - 队列**按提交时间顺序执行**，不支持管理员手动调整顺序。
- **用户操作**  
  - 普通用户可查看完整的打印队列（所有人任务），但**文件名仅自己和管理员可见**，他人只显示任务 ID。打印张数所有人可见。
  - 用户可取消自己**尚未开始打印**的任务（`queued`、`pending_review`）。
  - 用户可在“我的打印”中查看个人历史记录和实时状态。
- **管理员操作**  
  - 可取消**任意**未完成任务，可暂停/继续整个队列。
  - 暂停后当前正在打印的任务继续完成，后续任务不提交。
- **持久化**  
  - 所有任务（包括状态、文件路径、页数、提交者等）存入 SQLite。
  - 服务器重启后自动恢复未完成队列，并恢复暂停状态。
- **打印机状态与任务推送**  
  - 由于 HP P1106 是 host-based 打印机，**读取打印机实时状态（缺墨、卡纸）可能不可靠或不可行**。系统将做“尽力而为”尝试：
    - 若能通过 Windows 打印 API 获取打印机状态（如 `PRINTER_INFO_2`），则：
      - 实现 WebSocket 推送任务状态变化（包括打印机错误时暂停队列）。
      - 检测到缺纸、卡纸等错误时，自动暂停队列，待错误清除后继续。
      - 当前任务标记为 `printing`，完成后标记 `done`。
    - 若无法可靠获取状态，则：
      - 每次提交打印后直接标记为 `done`（或短暂 `printing` 后标记 `done`），不做复杂状态跟踪。
      - 不实现 WebSocket 推送（可选简单轮询）。
  - 最终方案在技术预研后确定，需求文档先保留 WebSocket 推送的预期，若实现困难则降级为轮询。
- **队列可见性**  
  - 所有用户均可看到完整队列列表（任务顺序、状态、页数），但只有管理员和任务所有者能看到文件名。

### 3. 用户系统

- **管理员**  
  - 初始管理员通过系统初次启动时指定学号，若该学号不存在则自动创建，默认密码同样为学号，强制首次修改。
  - 仅有一名管理员，但可在管理员面板**转让管理员**（选择另一用户升级为管理员，自己降级为普通用户）。
  - 管理员可设置自己的 QQ 号和学号（用于超额提示）→ 存入全局配置表。
  - 所有人都可以设置自己的QQ号，若被转让为管理员且填写了QQ号则自动使用这个QQ号用于超额提示。
- **用户创建与管理**  
  - 不开放注册，**仅管理员可批量导入**：上传 Excel 文件（第一列学号，无表头）。
  - 导入时自动为每个学号创建用户，默认密码为学号，首次登录强制修改密码。
  - 管理员可删除用户，删除时自动取消该用户所有未完成任务，并清理相关文件。
  - 管理员可修改任意用户密码（无强度限制）。
- **认证与 Cookie**  
  - 学号 + 密码登录，密码存储用 Argon2id。
  - 登录成功返回 HttpOnly Cookie，设置过期时间 1 年。每次访问刷新 Cookie 有效期。
  - Cookie 内包含随机 Token，服务端持久化 Token-用户映射（可存 SQLite）。
  - 无 CSRF 防护（内网信任环境）。
- **限额与额度查询**  
  - 用户登录后可看到今日已用张数、剩余额度。
- **历史记录**  
  - 所有完成的打印任务自动计入历史，记录：学号、文件名（原始名）、页数、打印时间、IP 等。
  - 历史记录及打印文件保存1年。
  - 用户可查看自己历史，管理员可查看所有用户历史及统计（按总张数降序排列）。
  - 统计粒度：总页数、总任务数。

### 4. 前端界面

- **技术**  
  Vue 3（组合式 API），`axios` 通信，`pdf.js` 预览，拖拽上传（如 `vue-upload-component` 或原生）。
- **页面模块**
  - 登录页 → 根据用户角色跳转
  - 普通用户端：
    - 提交打印：今日已用 / 限额。拖拽/点击上传多个文件，自动加入暂存列表，可预览每个文件（转为 PDF 后在前端显示），选择单双面打印（奇数/偶数/全部），确认后提交，然后自动跳转到我的打印。
    - 打印队列：卡片式展示所有任务，高亮自己的，显示状态，可取消自己的未开始任务。
    - 我的打印：个人历史与实时状态，列表带分页。
    - 个人设置：修改密码，查看管理员联系方式。
  - 管理员端（在以上基础上增加）：
    - 用户管理：导入 Excel、列表展示所有用户、删除用户、重置密码。
    - 队列管理：全部队列卡片，可取消任意任务、暂停/继续队列按钮。
    - 审核中心：待审核任务列表，可同意/拒绝，附理由。
    - 统计中心：按用户总打印张数降序排列，可导出简单报表或仅页面展示。
    - 系统设置：修改全局限额、设置管理员 QQ/学号、查看打印机状态（如有）、转让管理员。
- **风格**  
  小而美，简洁大方，少彩色，以白/灰/蓝为主调，卡片式布局，拖拽上传区域明显。
- **移动端** 不做适配。

### 5. 技术实现要点

- **后端 axum 服务器**  
  - RESTful API，JSON 交互。
  - WebSocket 端点（如 `/ws/queue`）用于队列状态推送。
  - 文档转换服务独立模块：使用 `std::process::Command` 调用 Word/Excel/PowerPoint 的 COM 对象（如 `win32com` 脚本或 PowerShell），或通过 LibreOffice 命令行（若安装）作为备用。
  - 文件存储：上传文件存于 `uploads/`，转换后 PDF 存于 `previews/`，按日期或任务 ID 组织，定期清理（已完成超过1年的任务文件）。
  - 定时任务：每日零点可重置限额计数（通过检查日期变化实现），若服务重启则补算。
  - 日志：使用 `tracing` 或 `log` 记录所有操作，包括打印提交、取消、管理员操作、错误等，便于审计。

- **SQLite 数据库表设计（简要）**
  - `users`: id, student_id, password_hash, role(admin/user), must_change_password, created_at
  - `sessions`: id, token, user_id, expires_at
  - `print_tasks`: id, user_id, file_name, original_name, page_count, status, odd_even (none/odd/even), submitted_at, completed_at, cancelled_by, review_reason
  - `daily_usage`: id, user_id, date, page_count (或实时从 tasks 计算)
  - `global_config`: key, value (限额, 管理员联系方式, 管理员学号等)
  - `audit_log`: id, user_id, action, details, ip, created_at (可选)

- **外部依赖**
  - Windows 环境需保证 Print Spooler 服务运行。
  - 若使用 Microsoft 365 转换，需编写一段 VBS 或 PowerShell 脚本，由 Rust 调用，并处理好超时和权限。

### 6. 待澄清/默认假设

- **假设**：打印机状态获取不可靠，最终可能降级为无实时状态推送，前端改用定时轮询（如 5 秒刷新队列状态）。文档中先保留 WebSocket 方案，实施时视情况调整。
- **假设**：COM 自动化转换 Word 时，服务需以有桌面交互权限的用户运行（Windows Service 需设置“允许与桌面交互”或通过特定账户启动）。此问题在部署时解决。
- **单日限额重置时间**：默认为北京时间 0:00，可配置。
- **文件清理策略**：已完成任务对应的源文件和 PDF 在1年后自动删除。
- **多文件上传后确认**：前端可逐文件预览并选择全部/奇数/偶数后，整体提交为多个独立任务。
- **待审核任务的处理**：管理员同意后任务变成 `queued` 并进入队列尾部；拒绝后变为 `cancelled`，并记录原因（前端可选展示给用户）。
